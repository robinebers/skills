# The Setup Process

Generate the right config so AI agents start in their isolated worktrees / environments with the same setup as the main repo: dependencies installed, env files in place, tool configs copied over.

1. Unless the context already provides an answer, explicitly ask the user which tool the setup is for: Cursor, Codex, or Conductor.

2. Map out the project (web, native, Rust, Swift, etc.) and make judgement calls on the right setup/install, run/actions, and teardown/archive commands.

**Not all projects use bun.** Some use `make`, `cargo`, `bun tauri dev`, `swift build`, `npm`, `pnpm`, `pip`, etc. Read the README, `package.json` scripts, top-level `Makefile`, or `Cargo.toml` to figure it out.

## What each tool reads

- **Cursor:** `.cursor/worktrees.json` (string path to `scripts/setup-worktree.sh`).
- **Codex:** `.codex/environments/environment.toml` ([setup].script delegates to the sh).
- **Conductor:** `conductor.json` at root (setup script invokes the sh).
- **Grok:** `~/.grok/worktrees.db` + auto-detection of the same sh.

## The shared pattern

Across all of them, the setup script usually does the same 3 things:

1. **Install dependencies** - run appropriate commands by project type, like Bun, Tauri, make, cargo, swift, etc. Read the README, `package.json`, `Makefile`, `Cargo.toml`, or similar to figure out what's right.
2. **Copy AI-tool config dirs** - `.agents`, `.claude`, `.codex`, `.cursor`, `.grok` from the source repo into the new environment (single source of truth lives in `scripts/setup-worktree.sh`).
3. **Copy env files** - `.env`, `.env.local`, `.env.development`, etc.

CRITICAL: the setup script always lives project-scoped, not global. All three (plus Grok) now delegate to the same `scripts/setup-worktree.sh` for true DRYness.

### The canonical script: `scripts/setup-worktree.sh`

The logic lives in one place: `scripts/setup-worktree.sh`. Every agent config is a thin delegator that invokes it.

**Contract:** the script reads two env vars, which each agent's delegator sets:

- `SOURCE_ROOT` â€” absolute path to the source repo (the original checkout)
- `WORKTREE_ROOT` â€” absolute path to the new worktree / workspace being set up

Full template (customize `{INSTALL_CMD}` and the env-file list for the project):

```bash
#!/usr/bin/env bash
set -euo pipefail

: "${SOURCE_ROOT:?SOURCE_ROOT must be set by the caller}"
: "${WORKTREE_ROOT:?WORKTREE_ROOT must be set by the caller}"

cd "$WORKTREE_ROOT"

# 1. Install dependencies
{INSTALL_CMD}

# 2. Copy AI-tool config dirs
for d in .agents .claude .codex .cursor .grok; do
    if [ -d "$SOURCE_ROOT/$d" ]; then
        mkdir -p "$WORKTREE_ROOT/$d"
        rsync -a "$SOURCE_ROOT/$d/." "$WORKTREE_ROOT/$d/"
    fi
done

# 3. Copy env files
for f in .env .env.local .env.development; do
    [ -f "$SOURCE_ROOT/$f" ] && cp "$SOURCE_ROOT/$f" "$WORKTREE_ROOT/$f" || true
done
```

After generating it, `chmod +x scripts/setup-worktree.sh` and commit it.

## Templates

Replace `{INSTALL_CMD}`, `{RUN_CMD}`, and `{ENV_FILE}` per project.

### Cursor

Cursor (official docs) supports a **string filepath** for `setup-worktree-unix` / `setup-worktree` (relative to `.cursor/worktrees.json`). This is the DRY way:

Place your canonical `scripts/setup-worktree.sh` at project root, then:

```json
{
    "setup-worktree-unix": "../scripts/setup-worktree.sh"
}
```

(Or use the generic `"setup-worktree"` key. Cursor runs the script with `$ROOT_WORKTREE_PATH` available.)

### Codex

The file is autogenerated by the Codex desktop app â€” don't hand-edit it lightly, but it's plain TOML and committed to git. The `[setup].script` is arbitrary bash; use the short delegation form so the real logic stays in `scripts/setup-worktree.sh`. The `[[actions]]` blocks each become a named shortcut in the Codex UI.

```toml
version = 1
name = "{PROJECT_NAME}"

[setup]
script = '''
#!/usr/bin/env bash
set -euo pipefail

SOURCE_ROOT="$CODEX_SOURCE_TREE_PATH" WORKTREE_ROOT="$CODEX_WORKTREE_PATH" "$CODEX_SOURCE_TREE_PATH/scripts/setup-worktree.sh"
'''

[[actions]]
name = "Dev Server"
icon = "run"
command = "{RUN_CMD}"

[[actions]]
name = "Build"
icon = "build"
command = "{BUILD_CMD}"
```

Common icons: `run`, `build`, `debug`, `logs`, `check`, `package`.

### Conductor

`conductor.json` lives at the repo root and gets shared with teammates so everyone gets the same workspace setup.

Fields:

| Field | Type | Description |
|---|---|---|
| `scripts.setup` | string | Command to run when setting up a new workspace. |
| `scripts.run` | string | Command to start the dev server. |
| `scripts.archive` | string | Command to run when archiving a workspace. |
| `runScriptMode` | `"concurrent"` \| `"nonconcurrent"` | Whether to kill in-progress run scripts before starting a new one. |

Template (already the DRY form â€” just invoke the canonical script):

```json
{
    "scripts": {
        "setup": "SOURCE_ROOT=\"$CONDUCTOR_ROOT_PATH\" WORKTREE_ROOT=\"$CONDUCTOR_WORKSPACE_PATH\" \"$CONDUCTOR_ROOT_PATH/scripts/setup-worktree.sh\"",
        "run": "{RUN_CMD}"
    }
}
```

## DRY delegation pattern (the recommended way for all four agents)

Generate `scripts/setup-worktree.sh` once (customize the install block for your package manager / framework). Then wire the three thin delegators so every agent uses the exact same logic:

**Cursor** (`.cursor/worktrees.json` at root):
```json
{
  "setup-worktree-unix": "../scripts/setup-worktree.sh"
}
```

**Codex** (`.codex/environments/environment.toml`):
```toml
[setup]
script = '''
#!/usr/bin/env bash
set -euo pipefail
SOURCE_ROOT="$CODEX_SOURCE_TREE_PATH" WORKTREE_ROOT="$CODEX_WORKTREE_PATH" "$CODEX_SOURCE_TREE_PATH/scripts/setup-worktree.sh"
'''
```

**Conductor** (`conductor.json` at root):
```json
{
  "scripts": {
    "setup": "SOURCE_ROOT=\"$CONDUCTOR_ROOT_PATH\" WORKTREE_ROOT=\"$CONDUCTOR_WORKSPACE_PATH\" \"$CONDUCTOR_ROOT_PATH/scripts/setup-worktree.sh\"",
    "run": "bun dev"
  }
}
```

**Grok** (when active): the worktree system already invokes the same sh via the DB + env detection.

Future changes (new dirs, smarter install, extra env files) only touch the one sh.

## Environment variables by platform

- **Cursor:** `$ROOT_WORKTREE_PATH` / (cwd is the new worktree, no separate destination var)
- **Codex:** `$CODEX_SOURCE_TREE_PATH` / `$CODEX_WORKTREE_PATH`
- **Conductor:** `$CONDUCTOR_ROOT_PATH` / `$CONDUCTOR_WORKSPACE_PATH`
- **Grok:** auto-detected via `~/.grok/worktrees.db` + `GROK_WORKSPACE_ROOT`

Vendors rename these. Always check the platform's docs before writing them.

## Usage

CRITICAL: The user is a non-technical, non-engineer. Avoid technical jargon and help them set up the correct setup scripts for their setup.

1. Identify the agent(s) â€” Cursor, Codex, Conductor (and Grok when active).
2. Detect project type (bun/npm, cargo, make, etc.) by reading package.json / README / Makefile.
3. Generate `scripts/setup-worktree.sh` (the single source â€” customize the install + env copy section for the project).
4. Generate the thin delegator for each requested agent (`.cursor/worktrees.json`, `.codex/environments/environment.toml`, `conductor.json`) using the DRY delegation pattern above.
5. Offer to also create supporting files (`.cursor/.gitignore` with `plans/`, `.claude/launch.json` with run configs from package.json, `.codex/config.toml` for MCP).
6. Remind them to `chmod +x scripts/setup-worktree.sh` and git commit so the whole team benefits.
7. Offer the same for any of the other agents.
